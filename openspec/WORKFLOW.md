## Getting Started — Fresh Project Bootstrap

### Step 0: Evaluate Project Scale

Before installing anything, assess where your project fits:

```
/mvp:evaluate-scale
```

This reads your project's filesystem signals (change count, module count,
repo count, existing spec coverage) and interviews you for non-detectable
signals (contributor count, AI agents, external vendors). It outputs a
structured recommendation:

- **Detected scale**: Sketch / Blueprint / Modular / Ecosystem
- **Confidence**: High / Medium / Low
- **Missing schemas**: What needs to be installed
- **Next action**: What to do next

Unlike `/opsx:explore` (which crystallizes vague ideas), `/mvp:evaluate-scale`
diagnoses the project itself and recommends the right schema setup.

### Step 1: Install the MVP-Schemas System

```bash
# From this repository, copy the mvp-schemas system into your project:
./scripts/copy-mvp-schemas.sh your-project/

# This copies: 7 schemas, config, reference docs, /mvp:evaluate-scale command
# Does NOT copy: stock overrides (opsx-* / openspec-*), project-specific content
```

If your project doesn't have `openspec/` yet, run `openspec init` first to
create the basic structure, then run the script.

> **Note on stock overrides**: This script does NOT copy the custom `opsx-*`
> commands or `openspec-*` skills from `.opencode/`. Each project should
> manage these independently — they may have project-specific customizations,
> and stock OpenCode versions may differ across setups. To install them
> manually, copy the relevant files from `.opencode/commands/` and
> `.opencode/skills/` in this repository.

To install everything manually:

```bash
# Schemas
cp -r openspec/schemas/project-profile    your-project/openspec/schemas/
cp -r openspec/schemas/epic               your-project/openspec/schemas/
cp -r openspec/schemas/vertical-slice     your-project/openspec/schemas/
cp -r openspec/schemas/spec-driven-enhanced your-project/openspec/schemas/
cp -r openspec/schemas/external-ingest    your-project/openspec/schemas/
cp -r openspec/schemas/code-to-spec       your-project/openspec/schemas/
cp -r openspec/schemas/project-upgrade    your-project/openspec/schemas/

# Config & reference
cp openspec/config.yaml                   your-project/openspec/
cp openspec/skills-profile.yaml           your-project/openspec/
cp openspec/WORKFLOW.md                   your-project/openspec/

# New mvp-schemas command & skill (optional — stock does not have these)
cp .opencode/commands/mvp-evaluate-scale.md  your-project/.opencode/commands/
cp -r .opencode/skills/mvp-evaluate-scale    your-project/.opencode/skills/
```

### Step 2: Explore and Assess Fit-Level (If Needed)

```
/opsx:explore
```

If you already ran `/mvp:evaluate-scale` and know your scale, you may
skip this step. Use `/opsx:explore` when you have a **specific idea
or initiative** to crystallize — it concludes with a fit-level
recommendation (project-wide / new-epic / vertical-slice / simple-change)
that tells you which schema to use for that initiative.

The AI reads `openspec/config.yaml` context, which contains:
- Project Scale Selection guide (Sketch → Blueprint → Modular → Ecosystem)
- Fit-level recommendation logic (project-wide / new-epic / vertical-slice / simple-change)
- Exploration output template

The explorer concludes with a concrete recommendation:

```
## What We Figured Out
**The idea**: A real-time collaborative whiteboard app
**Scope analysis**: Multiple components (canvas engine, sync protocol,
  auth, UI), 2+ repos, needs vendor SDK for drawing primitives
**Recommended fit level**:
  ☑ new-epic — spans multiple independent testable streams
**Recommended scale**: Modular
**Recommended schemas**: project-profile + epic + vertical-slice + spec-driven-enhanced
**Recommended next step**: Create project-profile to bootstrap identity and governance,
or run `/mvp:evaluate-scale` first if you need scale confirmation.
```

### Step 3: Bootstrap Project Identity

```
openspec new change init-profile --schema project-profile
```

This generates into `openspec/project/` (permanent, never archived):
- `profile.md` — auto-scanning status dashboard
- `constitution.md` — hard governance constraints
- `standards.md` — advisory coding conventions
- `architecture.md` — domain canvas

### Step 4: Follow the Scale-Appropriate Path

```
Sketch user     → stock spec-driven (skip profile, add features directly)
Blueprint user  → project-profile + spec-driven-enhanced
Modular user    → + epic + vertical-slice
Ecosystem user  → + external-ingest + code-to-spec
```

### Explorer Is Not Just Step 0

`/opsx:explore` can be invoked ANYTIME, at ANY scale, mid-project.
It always reads the current config.yaml context, so it always knows
about your project's scale system. Use it whenever:
- You have a vague new idea and need to assess where it fits
- An existing change feels bigger than expected — re-assess fit-level
- You discover a new integration that needs vendor docs

## How the Explorer Knows About Scales

The stock OpenSpec explorer (`/opsx:explore`) is a built-in skill.
It does NOT have hardcoded knowledge of our custom scale system.

**How it works**: When you invoke `/opsx:explore`, OpenSpec injects
`openspec/config.yaml` context into the AI prompt. Since our config.yaml
contains the full scale selection logic, fit-level decision tree,
and exploration output template, the explorer AI reads this and
naturally follows the custom flow.

**Key rule**: The config.yaml context IS the orchestrator. You don't need
a meta-schema that rules over all 7 schemas. The context in config.yaml
guides the explorer, and each schema's instructions guide its own workflow.
The schemas compose through cross-referencing (epic links to slices,
slices link to changes), not through a parent schema.

## Evaluate-Scale Command vs Explorer

`/mvp:evaluate-scale` and `/opsx:explore` are complementary but serve
different entry points:

| | `/mvp:evaluate-scale` | `/opsx:explore` |
|---|---|---|
| **Purpose** | Diagnose the project's scale | Crystallize a vague idea |
| **When to use** | New project onboarding, mid-project re-evaluation, scale audit | "I have an idea but need to think it through" |
| **Input** | Project filesystem state | A fuzzy idea, problem, or question |
| **Output** | Scale recommendation + missing schemas | Crystalized idea + fit-level recommendation |
| **Reads/writes?** | Read-only (never modifies files) | Read-only (never implements code) |
| **Who runs first** | First (for scale assessment) | Second (for specific initiative planning) |

**Typical flow:**

```
/mvp:evaluate-scale
→ "You're at Modular scale. Schemas installed correctly."

/opsx:explore
→ "I want to add real-time chat."
→ "Fit-level: new-epic. Create an epic and slices."

## Architecture Philosophy

### Scale Is About Ambition, Not Headcount

A solo developer orchestrating 5 AI agents faces the same coordination
challenges as a 5-person team. Each AI agent is an independent work
stream that needs bounded context, clear contracts, and coordination.
Project scale is determined by **how many independent work streams must
coordinate** and **how much work the goal implies** — not by how many
humans are in the room.

### Design Constraint: No CLI Code Changes

All seven schemas use OpenSpec's existing customization system:
`schema.yaml`, templates, `requires` dependency graph, `apply.tracks`.
No modifications to the OpenSpec CLI source code are required or assumed.
Every schema is a configuration artifact, not a code change.

### Why a Schema Family (Not One Monolithic Schema)

A single schema with 15+ optional artifacts was rejected because:
- Optional artifacts weaken `requires` dependency enforcement
- AI has too many degrees of freedom with optional artifacts
- A single "enterprise-only" schema is overkill for small projects
  and violates OpenSpec's "easy not complex" principle

The family approach lets projects adopt schemas incrementally:
Sketch users see 1 schema; Blueprint users see 2; Modular users see 4.

## Project Scale Selection

```
Sketch     — A simple idea, vague scope, 1-2 concurrent work streams.
             Schemas: stock spec-driven
             Explorer: basic scale assessment only

Blueprint  — Scope is crystallizing. Need identity, governance, standards.
             Schemas: project-profile + spec-driven-enhanced + skills-profile
             Explorer: full fit-level recommendation

Modular    — Multiple independent workstreams. Need epic/slice decomposition.
             Schemas: + epic + vertical-slice + domain-map
             Explorer: full fit-level recommendation

Ecosystem  — Multiple repos, teams, or external vendors. Full SDD governance.
              Schemas: + external-ingest + code-to-spec
              Explorer: full fit-level recommendation

## Project Scale Assessment Signals

When `/mvp:evaluate-scale` runs, it combines auto-detected filesystem
signals with user interview signals to determine the recommended scale.

### Auto-Detectable Signals

| Signal | Detection Method | Used For |
|--------|-----------------|----------|
| Active changes count | Count `openspec/changes/` dirs (non-archive) | Work stream estimate |
| Active epics count | Count `openspec/epics/` dirs | Work stream estimate |
| Repo count | `go.mod`/`Cargo.toml`/`package.json` at root vs `.gitmodules` | +1 modifier if polyrepo |
| Bounded context estimate | Top-level `src/*/`, `pkg/*/`, `internal/*/` dirs | +1 modifier if > 8 |
| External API specs | Files containing `openapi:`/`swagger:`/`paths:` | +1 modifier if > 3 |
| Has project-profile | `openspec/project/` exists | -1 modifier (already governed) |
| Has vendor-specs | `openspec/vendor-specs/` exists | +1 modifier (already ingesting) |
| Undocumented code ratio | Spec files vs source module ratio | +1 modifier if high |
| Current config schema | `openspec/config.yaml` schema field | Baseline comparison |

### Manual Signals (Asked During Assessment)

| Signal | Why It Matters |
|--------|---------------|
| Human contributors | More people = more coordination |
| AI agents in use | Each AI agent is an independent work stream |
| External vendors | Each vendor adds contract management overhead |
| Multi-team vs single-team | Cross-team coordination is a +1 modifier |
| Current setup satisfaction | "Too heavy?" → downgrade. "Too loose?" → upgrade |

### Decision Logic

**Primary signal**: Estimated work stream count:
- 1-2 → Sketch
- 2-5 → Blueprint
- 5-12 → Modular
- 12+ → Ecosystem

**Modifier signals** adjust the candidate up or down on the scale ladder:

| Condition | Adjustment |
|-----------|------------|
| Polyrepo (>= 2 repos) | +1 |
| External vendors >= 3 | +1 |
| Multi-team | +1 |
| Bounded contexts > 8 | +1 |
| High undocumented code ratio | +1 |
| External API specs > 3 | +1 |
| Has project-profile | -1 |
| Single team AND monorepo | -1 |
| Solo dev + 1 AI agent | -1 |
```

## Scale Upgrade Paths

### Sketch → Blueprint

```
/mvp:evaluate-scale        # Confirm you've outgrown Sketch
openspec new change upgrade-sketch-to-blueprint --schema project-upgrade
→ Scans existing specs, config, changes
→ Bootstraps project-profile artifacts (profile, constitution, standards, architecture)
→ Forks spec-driven → spec-driven-enhanced
→ Updates config.yaml with enhanced rules
→ Verification: confirms AI reads constitution and standards
```

### Blueprint → Modular

```
/mvp:evaluate-scale        # Confirm you've outgrown Blueprint
openspec new change upgrade-blueprint-to-modular --schema project-upgrade
→ Analyzes active changes, spec complexity
→ Creates epic structure from existing vertical slices
→ Adds domain-map if multi-repo
→ Updates config.yaml with epic/slice schemas
→ Verification: confirms epic decomposition works
```

### Modular → Ecosystem

```
/mvp:evaluate-scale        # Confirm you've outgrown Modular
openspec new change upgrade-modular-to-ecosystem --schema project-upgrade
→ Adds external-ingest and code-to-spec schemas
→ Creates openspec/vendor-specs/ directory
→ Updates skills-profile.yaml with ingest/reverse-engineer entries
→ Verification: confirms ingestion pipeline works
```

## End-to-End Walkthrough: Modular-Scale Project

Example: building a "WeChat clone" for learning distributed systems.

### Step 0: Evaluate Scale and Explore

```
/mvp:evaluate-scale
→ "Modular scale confirmed. Schemas project-profile + epic + vertical-slice
   + spec-driven-enhanced recommended."

/opsx:explore
→ "I want a WeChat clone. Not sure where to start."
→ AI explores: existing codebase, similar projects, architecture options
→ Crystallizes: "Modular-scale project. Need project-profile,
  epic for core initiative, vertical slices for each milestone."
→ Fit-level: new-epic for "Core Messaging"
```

### Step 1: Bootstrap Project Identity

```
openspec new change init-profile --schema project-profile
→ Creates openspec/project/profile.md (auto-scans existing specs)
→ Creates openspec/project/constitution.md (hard constraints)
→ Creates openspec/project/standards.md (advisory: Go style, gRPC patterns)
→ Creates openspec/project/architecture.md (domain canvas)
→ Creates openspec/project/TODO.md (emergent ideas inbox)
```

### Step 2: Create Epics

```
Manually create: openspec/epics/2025-05-messaging/epic.md
→ Vision: "Real-time messaging between users"
→ Slices:
  Slice 1: Infrastructure — "Project scaffold compiles and CI passes"
  Slice 2: Feature — "User can send and receive text messages"
  Slice 3: Feature — "User can see online/offline status"
→ openspec/epics/2025-05-messaging/TODO.md (future ideas)
```

### Step 3: Plan Each Slice

```
openspec new change slice-messaging-scaffold --schema vertical-slice
→ slice-proposal.md: compile + CI as doD
→ change-manifest.md:
    change: be-scaffold → backend-service (Go, gRPC server)
    change: fe-scaffold → web-dashboard (React, WebSocket client)

openspec new change slice-send-receive --schema vertical-slice
→ change-manifest.md:
    change: be-messaging-api → backend-service
    change: fe-chat-ui → web-dashboard
    contract: gRPC SendMessage(Message) → MessageResponse
```

### Step 4: Implement Each Change

```
openspec new change be-messaging-api --schema spec-driven-enhanced
→ proposal (parent: slice-send-receive) → specs ([REQ-MSG-001])
→ design (ADR: gRPC over WebSocket) → tasks (blast radius: backend/msg/*)
→ verification (traceability matrix, git diff compliance)
→ apply → archive
```

Mid-implementation triage:
> "be-messaging-api/TODO.md has 'add message persistence'. Add as task or defer?"

### Step 5: Reverse-Engineer Undocumented Code (if needed)

```
openspec new change reverse-db-layer --schema code-to-spec
→ reverse-engineering.md: reads backend/db/* → BDD specs
→ fidelity-report.md: 85% covered, 3 uncovered error paths
→ Creates openspec/vendor-specs/db-layer/spec.md [REQ-CODE-001..003]
→ Recommends: create change to add tests for uncovered paths
```

### Step 6: Ingest External Docs (if applicable)

```
openspec new change ingest-push-notification-vendor --schema external-ingest
→ source-manifest: VNDR-001, Swagger 3.0, PushCorp API v2
→ spec-extraction: [REQ-VENDOR-001] Push notification dispatch
→ gap-analysis: missing retry logic, missing SLA monitoring
→ Creates openspec/vendor-specs/pushcorp/spec.md
→ Recommends: create change to add retry + monitoring
```

### Step 7: Keep Profile Current

```
Run /opsx:verify project-profile periodically
→ profile.md auto-updates status dashboard
→ Active Epics: 2025-05-messaging (2/3 slices done)
→ AI surfaces: "project/TODO.md has 3 items. Triage?"
```

## Risks and Trade-offs

| Risk | Mitigation |
|------|------------|
| Schema sprawl — 7 schemas may overwhelm | Scale table guides adoption; `/mvp:evaluate-scale` diagnoses exact needs |
| Auto-derived status stale | profile.md shows "Last auto-derived: <date>" |
| Epics outside CLI tooling | Markdown files — scanned for auto-derivation |
| Blast radius too coarse/fine | Task-level granularity; verification.md catches violations |
| Shadow specs become stale | fidelity-report.md flags confidence; periodic re-ingestion |
| project-upgrade misses nuances | AI-assisted but human-reviewed; verification is the gate |
| TODO.md becomes cruft | Periodic triage prompts; items >90 days flagged dormant |
| Fan-out fragments context | Threshold: ~500 lines; `_index.md` maintains coherence |
| Community skills unavailable/broken | Skills are suggestions; AI gracefully skips unavailable |
| Solo dev misses guidance designed for teams | Scale is ambition-not-headcount; solo + AI = virtual team |

## Open Questions

1. **CLI support for epics**: Should `openspec epic new <name>` exist?
   Manual creation works without CLI changes. Upstream contribution opportunity.

2. **profile.md refresh trigger**: On `openspec list`? On archive?
   Explicit invocation? Current design: explicit with stale-date watermark.

3. **Vendor-spec promotion**: Opt-in mechanism to promote shadow spec to
   authoritative? Manual promotion + human review is safest default.

4. **Cross-slice dependencies**: How to track that Slice 2 needs a refactor
   from Slice 1? change-manifest in Slice 2 references Slice 1 as prerequisite.

5. **TODO.md deduplication**: Same idea appears in change AND epic TODO.md?
   AI triage should detect near-duplicates and suggest merging.

6. **suggested_skills as official extension**: Currently a convention.
   Formal `suggested_skills` field in schema YAML would require upstream
   contribution to OpenSpec CLI.

7. **code-to-spec vs external-ingest overlap**: Both output to `vendor-specs/`.
   Shared output format, different source handling and confidence characteristics.
